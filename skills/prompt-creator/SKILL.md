# prompt-creator

帮助用户创建、优化和沉淀系统提示词的skill。

## 概述

prompt-creator 旨在帮助用户从实际使用中提炼经验，持续优化系统提示词。类似"代码审查"但针对系统提示词工程。

## 为什么需要这个skill

系统提示词（AGENTS.md）是影响AI行为的核心配置。随着使用深入，会积累大量经验：

- **偏好**：如"简洁回复"、"不用markdown"
- **规则**：如"优先使用edit"
- **最佳实践**：如"分层级管理配置"

这些经验如果只停留在对话中，会随会话结束而丢失。这个skill帮助您将这些经验沉淀到系统提示词中，实现：
1. 避免重复解释相同偏好
2. 积累可复用的最佳实践
3. 持续优化AI行为

<context>
为什么沉淀到AGENTS.md很重要：系统提示词在每次新会话开始时加载，沉淀后的规则会被永久记忆，避免每次都需要重新解释相同的需求。这让AI能够持续学习用户的偏好，提供更个性化的服务。
</context>

## 三原则应用指南

### 分类：L1通用 vs L2工具特定

本skill内容按以下原则分类：

| 分类 | 说明 | 内容来源 |
|------|------|----------|
| **L1 通用基础** | 适用于所有LLM工具的提示词工程原则 | 本skill核心章节 |
| **L2 工具特定** | 特定工具（如OpenCode）的机制知识 | `references/OpenCode机制.md` |

**应用规则**：
- 优先使用L1通用知识解决问题
- 只有当用户明确使用特定工具时才加载L2知识

## 典型场景（按分级）

### Basic：提炼系统提示词
<skill_tier>Basic</skill_tier>

用户完成一系列会话后，想要沉淀经验

**推荐触发**：
- "是否有可提炼的系统提示词"
- "把这次的经验沉淀下去"
- "我想把偏好固定下来"
- "修改全局系统提示词"

**处理流程**：
1. 快速分析会话上下文，聚焦识别近期明确的用户偏好、重复规则和高频反馈
2. 与现有agents.md对比，检测重复或需要更新的部分
3. 生成修改建议，包含XML格式的系统提示词片段
4. 让用户确认目标层级和具体文件位置
5. 辅助执行修改
6. 提醒用户重启工具或执行热加载

### Intermediate：审查现有提示词
<skill_tier>Intermediate</skill_tier>

用户请求审查或优化现有配置

**推荐触发**：
- "看看当前项目的提示词是否有问题"
- "检查下配置有没有问题"
- "审查AGENTS.md"

**扩展触发**：
- "检查引用文件"
- "同步引用"
- "审查AGENTS.md及引用"

**处理流程**：
1. 确定目标AGENTS.md文件位置（根据上下文或询问用户）
2. 对照最佳实践快速审查关键点
3. 优先识别影响核心功能的明显问题
4. 检查引用文件是否需要同步更新：
   - 解析@引用路径
   - 验证引用文件存在性
   - 检查引用内容一致性
5. 生成优化建议报告（含主文件+引用文件）
6. 用户确认后辅助修改
7. 如有引用文件更新，一并处理
8. 提醒用户重启/热加载

### Advanced：技能自我验证
<skill_tier>Advanced</skill_tier>

当用户要求验证技能自身是否符合最佳实践时触发。

**触发**：用技能验证自身是否符合最佳实践、审查这个skill的配置

**处理**：
1. 识别为自我验证场景
2. **引用官方最佳实践链接**（仅此场景）
3. 对照审查技能内容，生成报告

## L1：通用最佳实践（L1 通用基础）

### 核心原则

1. **明确性**：指令应清晰明确，避免歧义
2. **上下文**：提供指令背后的动机，帮助LLM理解目标
3. **示例**：用"效果更好"的示例替代"效果较差"的示例
4. **安全性**：考虑操作可逆性，要求确认风险操作
5. **效率**：聚焦关键问题，避免过度分析；必要时告知用户深入分析的范围
6. **自主性**：明确默认行为倾向，确保主动执行而非仅提供建议

<default_to_action>
默认情况下，实施更改而不是仅仅建议它们。当用户意图不明确时，推断最有用的可能操作并继续执行，使用工具来发现任何缺失的细节而不是猜测。如果用户明确要求修改或优化，则直接执行而非仅提供建议。
</default_to_action>

### 层级判断规则（L1 通用）

#### 推荐的层级划分

- 最外层：跨项目通用配置
- 中间层：领域/工作区配置
- 最内层：项目特定配置

#### 优先级规则
内层配置可以覆盖外层配置。

#### 内容类型与层级对应

| 内容类型 | 建议位置 |
|----------|----------|
| 跨项目通用偏好 | 最外层 |
| 特定领域规则 | 中间层 |
| 单项目特定规则 | 最内层 |
| 临时经验 | 不建议沉淀 |

#### 如何确定当前层级

1. **从上下文中获取**：读取用户的AGENTS.md获取其定义的层级
2. **从路径推断**：根据当前工作目录的AGENTS.md文件位置判断
3. **询问用户**：如不明确，直接询问用户

## 工作流程

### 1. 会话沉淀流程

```
1. 接收用户请求
2. 分析当前会话上下文：
   - 用户的明确偏好（如"简洁回复"）
   - 重复出现的规则（如"不要用markdown"）
   - 成功的问题解决方案
   - 用户的纠正/反馈
3. 搜索并读取现有AGENTS.md（全局/工作区/项目层级），**必须实际读取文件内容**
4. 检测重复：
   - 搜索是否有相似的偏好/规则已存在
   - 标记需要更新的现有规则
5. 判断目标文件是否存在：
   - 若存在 → 生成修改建议
   - 若不存在 → 读取配置架构，推荐合适的层级架构模式
6. 生成建议：
   - 提取的内容（XML格式）
   - 建议添加位置
   - 理由说明
   - **对比结果**（列出发现的重叠或冲突）
7. 呈现给用户确认
8. 用户确认后辅助修改
9. 提醒用户重启/热加载
```

### 2. 审查优化流程

```
1. 确定目标文件（根据上下文确定层级，或询问用户）
2. 读取 agents.md 内容
3. 验证问题是否真实存在（观察实际行为）
4. 对照最佳实践清单逐项审查
5. 区分设计决策与实际问题
6. 生成审查报告：
   - 问题列表
   - 优化建议
   - 优先级排序
7. 呈现给用户
8. 用户确认后辅助修改
9. 提醒用户重启/热加载
```

## 输出格式

详细输出格式模板请读取 `references/输出格式.md`

### 核心要素

- 明确区分"建议内容"与"理由说明"
- 列出目标层级选项，让用户确认
- 修改后提醒用户重启/热加载

## 注意事项

1. **用户主导**：始终让用户确认修改位置和内容，不自动修改
2. **避免重复**：添加前检查现有agents.md是否已有类似内容
3. **适度沉淀**：并非所有经验都值得沉淀，避免过度工程化
4. **清晰解释**：说明为什么某内容值得沉淀及其影响
5. **风险警示**：修改 AGENTS.md 是**高风险操作**
   - 外层配置影响范围大，内层影响范围小
   - 建议修改前备份（git），修改后测试验证
6. **层级实践**：
   - 首先确认用户的配置架构（读取用户的AGENTS.md）
   - 外层定义基础规范，内层进行具体化
   - 避免在非定义层重复已存在的规则
