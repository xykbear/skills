# 系统提示词审查清单

## 核心原则

1. **明确性**：指令应清晰明确，避免歧义
2. **上下文**：提供指令背后的动机，帮助LLM理解目标
3. **示例**：用"效果更好"的示例替代"效果较差"的示例
4. **安全性**：考虑操作可逆性，要求确认风险操作
5. **自主性**：明确默认行为倾向，确保主动执行而非仅提供建议

## 常见优化维度

| 维度 | 检查点 |
|------|--------|
| 明确性 | 指令是否清晰无歧义 |
| 上下文 | 是否提供动机/背景 |
| 安全性 | 风险操作是否需要确认 |
| 自主性 | 行为倾向是否明确 |
| 状态管理 | 是否有进度跟踪机制 |
| 边界限制 | 是否明确禁止行为 |
| 引用一致性 | @引用路径是否有效 | 引用文件是否存在 |
| 引用同步 | 引用内容是否与主题一致 | 是否有遗漏的引用 |

## 审查流程

1. 确定目标文件位置和层级
2. 逐项对照清单审查
3. 验证问题是否真实存在（观察实际行为）
4. 区分设计决策与实际问题
5. 生成优化建议
6. 按优先级排序输出

## 审查判断标准

### 验证优先原则

**必须先验证再提出问题**：
- 观察实际行为而非基于假设
- 理解不同层级的配置有不同职责
- 确认问题的真实存在

### 区分设计决策与问题

| 表面现象 | 实际判断 | 处理方式 |
|----------|----------|----------|
| 多处风险提示 | 可能是静态+动态分层保护 | 验证是否各有其上下文意义 |
| 规则层与实现层内容相似 | 可能是分层设计 | 理解"不准"vs"how"的区别 |
| 术语未解释 | 模型能理解即可 | 不强制要求解释 |
| 缺少索引/目录 | 可能有触发条件替代 | 验证触发条件是否有效 |

### 常见误判类型

1. **机械对照清单**：只看"有没有"，不看"为什么这样设计"
2. **过度工程化**：把正常设计误判为"问题"
3. **缺乏系统理解**：不理解实际行为

## 引用文件检查规则

### 自动检查时机
- 修改AGENTS.md后自动提示引用文件检查
- 审查流程中自动触发

### 显式触发
- "检查引用文件"
- "同步引用"
- "审查AGENTS.md及引用"

### 检查逻辑
1. 解析AGENTS.md中的@引用路径
2. 验证引用文件是否存在
3. 检查引用文件内容是否与引用场景一致
4. 生成同步建议
